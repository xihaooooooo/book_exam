# 全局最优审查报告

## 审查目标

`多学科多模态扩展计划.md` —— 将纯文本学习系统通过 L0-L5 六层逐步扩展到支持公式、图形、交互式题型。

## 发现的问题

### 问题 1：knowledge 存储介质分裂

- **局部视角下的合理性**：图片天然是二进制文件，存文件系统最直接；文本存在 SQLite 里，各管各的。
- **全局视角下的问题**：`sections.db` 新增 `images_json` 列存图片路径，路径指向 `cache/images/`。当数据库被移动/备份/复制时，图片引用全部断裂。图片描述（多模态 LLM 生成）又存回 text 列——一段知识点在三个地方（原始文字、图片路径、图片描述），更新一处另外两处不会同步。
- **维度**：数据一致性
- **严重程度**：中
- **替代思路**：
  - 图片描述和原始 OCR 文字合并为 `text` 列，格式化为 `[图 3-4 描述: ...]` 插入对应章节位置，而非分列存储
  - 图片文件用 content-hash 命名（SHA256 前 8 位），不依赖目录结构，移动数据库时至少能校验缺失
  - 或者在 `sections.db` 里直接用 BLOB 存小图（<1MB），大图才走文件引用

### 问题 2：question 的数据结构会被"类型膨胀"撑破

- **局部视角下的合理性**：每个新能力加一个新字段——`image_ref` for L2、`mermaid_code` for L3、`canvas_config` for L4。每个字段独立清晰。
- **全局视角下的问题**：`questions_*.json` 里的题目的字段会从现在的 10 个膨胀到 20+，且大部分字段对大部分题目为空（选择题不需要 diagram，Canvas 题不需要 options）。继续膨胀下去，前端渲染逻辑会变成 if-else 地狱：先判断有没有 `mermaid_code`，再判断有没有 `canvas_config`，再判断有没有 `image_ref`……
- **维度**：短期 vs 长期
- **严重程度**：中
- **替代思路**：
  - 设一个统一的 `media` 字段，`{"type": "image"|"mermaid"|"latex"|"canvas", "content": "...", "config": {...}}`
  - 前端就不需要逐字段判断，走到 `media` 分支后按 `type` 选渲染器
  - 这样做的好处：加新题型只需要加一种 `media.type`，不改数据结构 schema

### 问题 3：L4 "每学科一个 Canvas" 把复杂度摊给了前端

- **局部视角下的合理性**：每种题型的交互完全不同，独立组件最灵活。
- **全局视角下的问题**：按计划，数据结构 Canvas、电路 Canvas、几何 Canvas 是独立组件。每加一门学科，前端多一个组件，后端多一种判题逻辑，数据库多一种存储格式。三门学科 = 三套东西，十门学科 = 十套东西。没有统一的"交互式题型"抽象。
- **维度**：耦合方向 + 短期 vs 长期
- **严重程度**：中
- **替代思路**：
  - L4 做之前先定义"交互式题型的通用协议"：`{type, config_json, expected_answer_json, student_answer_json}`
  - Canvas 组件不存业务逻辑，只负责"给定 config 渲染 + 捕获用户操作 → 输出 answer_json"
  - 判题走通用管线：比较 `expected_answer_json` 和 `student_answer_json`（结构化的 diff）
  - 这样前端加新 Canvas 只是一层皮，后端不变

### 问题 4：L1 的 LaTeX "停止清洗"和画像冲突

- **局部视角下的合理性**：LaTeX 公式应该保留原始格式，让 KaTeX 在前端渲染。
- **全局视角下的问题**：`_load_section_titles` 在画像展示时清洗 LaTeX（去掉 `\mathrm{}`、`\mathbf{}` 等命令）。L1 说"不做清洗"，但画像页展示章节标题时如果保留原始 LaTeX——`"4.1 $\mu \mathrm{C} / \mathrm{OS}-\mathrm{II}$ 的中断"`——前端没装 KaTeX 的地方会显示乱码。
- **维度**：认知负担 + 数据一致性
- **严重程度**：低
- **替代思路**：
  - 清洗逻辑保留，但输出"纯文本标题"和"富文本标题"两个字段
  - 画像页用纯文本标题，答题区用富文本渲染
  - 或者前端全局挂 KaTeX，所有文本都过一遍渲染，LaTeX 留下，纯文本不受影响

---

## 综合评估

- **全局风险等级**：低
- **值得保留的设计**：
  - L0→L5 分层路线，按学科增量加——方向正确
  - L2 "图片转文字描述"不动前端——务实
  - 不做通用图形引擎——明智
  - L1 LaTeX 优先——ROI 最高的一步
- **建议重新考虑的设计**：
  - 图片路径存 DB（问题 1）——用 content-hash 或 BLOB 替代裸路径
  - questions 字段膨胀（问题 2）——用统一的 `media` 字段收口
  - L4 的 N 个 Canvas（问题 3）——先定义"交互题通用协议"
- **补充约束**：
  - L1 做完后，所有新增题目字段必须反思"能不能放进 media 统一字段"
  - L2 做之前，明确"图片描述"的存储位置（合并进 text 列 vs 独立列），并写进 CLAUDE.md
  - L4 做之前，先写一份"交互式题型协议"的 plan 文档
