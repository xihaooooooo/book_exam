# 多学科多模态扩展计划

## 现状

系统是**纯文本管道**：教材 OCR → 文本知识点 → LLM 生成文字题 → 文字判题。不处理任何非文本内容。

## 问题

- 理工科大量题目依赖公式、图形、图表
- 不同学科对"图"的需求完全不同——没有通用的图形引擎
- 如果只做纯文本，系统只能覆盖文科和概念题

## 分层路线

```
L0 纯文本    [已有] 文科、概念题
L1 公式      [马上] 理工科计算题  ← 下一步
L2 静态图    [随后] 看图分析题
L3 生成图    [选做] 流程/结构题
L4 交互图    [远期] 几何/电路
L5 智能判图  [远期] 多模态判题
```

每多做一级，工程复杂度翻一倍。按学科增量加，不做全科目通用。

---

## 跨层约定：统一 media 字段

所有 L1-L4 的新能力，不各自往 `questions_*.json` 加字段。统一用一个 `media` 对象收口：

```json
{
  "stem": "根据下图回答...",
  "options": ["A", "B", "C", "D"],
  "media": {
    "type": "image",
    "src": "a3f2b1c0.png",
    "content": "...",
    "config": {}
  }
}
```

| media.type | 适用层 | 含义 |
|---|---|---|
| `latex` | L1 | 题目含 LaTeX 公式 |
| `image` | L2 | 教材静态图 |
| `mermaid` | L3 | LLM 生成的流程/结构图 |
| `canvas` | L4 | 交互式画布 |

**前端只需一个分支**：

```javascript
if (q.media) {
  renderers[q.media.type](q.media)  // 按 type 选渲染器
}
```

加新类型只加一个 renderer，不改数据结构。纯文本题 `media` 为 null，向后兼容。

- 教材入库：parse.py OCR → sections.db (text 列)
- 出题：LLM 根据教材文本生成纯文字题目
- 答题：前端 input/textarea

**覆盖学科**：文科概念题、操作系统概念题、编程语言语法

## L1：公式支持（下一步）

### 需要改的

| 层 | 改动 | 工作量 |
|---|---|---|
| **存储** | `sections.db` 的 text 保留原始 LaTeX | 无（已保留） |
| **出题** | LLM prompt 要求公式用 `$...$` 内联 / `$$...$$` 块级 | 1 行 prompt |
| **题目存储** | `questions_*.json` 的 media 字段承载 LaTeX 标记 | media.type: "latex" |
| **判题** | LLM 比对时忽略 LaTeX 格式差异 | prompt 微调 |
| **前端** | 引入 KaTeX，全局渲染所有文本中的 `$...$` | ~5 行 JS |
| **画像** | `_load_section_titles` 产出两份：`section_titles`（纯文本，画像用）和 `section_titles_rich`（原始 LaTeX，答题区用） | 改 5 行 |

### 画像冲突解决

`_load_section_titles` 目前清洗 LaTeX 命令（`$\mu \mathrm{C}$` → `μC`）。L1 之后不改这个逻辑——画像页用纯文本版本。同时多输出一份 `section_titles_rich` 保留原始 LaTeX，供答题区 KaTeX 渲染。

### KaTeX 接入

```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/katex@0.16/dist/katex.min.css">
<script src="https://cdn.jsdelivr.net/npm/katex@0.16/dist/katex.min.js"></script>
<script>
// 渲染页面中的 LaTeX
document.querySelectorAll('.stem, .option, .explanation, .reason').forEach(el => {
  el.innerHTML = el.innerHTML.replace(/\$\$([^$]+)\$\$/g, (_, tex) => katex.renderToString(tex, {displayMode: true}));
  el.innerHTML = el.innerHTML.replace(/\$([^$]+)\$/g, (_, tex) => katex.renderToString(tex));
});
</script>
```

### 验收

- 出题 prompt 里加 "数学表达式用 LaTeX 语法：$...$ 内联，$$...$$ 单独成行"
- 生成一道带公式的题，前端渲染出正确的数学表达式
- 判题忽略 $(a+b)^2$ 和 $(a+b)^2$ 的空格差异

### 覆盖学科

数学、物理、化学计算、信号系统、电路分析——所有需要数学表达式的学科。

## L2：静态图（随后）

### 目标

教材里的图能出现在题目里，学生能在答案中引用图信息。

### 需要改的

| 层 | 改动 |
|---|---|
| **教材入库** | parse.py 提取图片 → 以 content-hash (SHA256 前 8 位) 命名存到 `cache/images/`；图片描述合并写入 `sections.db` 的 `text` 列，格式为 `[插图: a3f2b1c0.png] 描述文字...`，不单独分列 |
| **出题** | chief_editor 已有的 `search_keyword` 和 `get_section_text` 就能搜到图片描述，不需要新 tool |
| **题目存储** | 使用统一 `media` 字段：`{type: "image", src: "a3f2b1c0.png", description: "..."}` |
| **前端** | 题目区 `<img>` 展示图片 |
| **判题** | 图片描述附加到 LLM 判题 prompt 上下文中 |

### 图片转文字描述（降低门槛）

入库时用多模态 LLM 把图片转成文字，**合并写入 `text` 列**，不单独分列：

```
text 列（第 3 章原文）:
  "任务有 5 个状态...

  [插图: a3f2b1c0.png]
  该图描述了任务状态转换：睡眠态→就绪态(OSTaskCreate)、
  就绪态→运行态(调度器)、运行态→等待态(请求资源)..."
```

这样做的好处：
- chief_editor 不需要新 tool——已有的 `search_keyword` 和 `get_section_text` 就能搜到图片描述
- 图片文件名用 content-hash（如 `a3f2b1c0.png`），拷到哪都不会重名
- 数据库移动时能校验图片是否存在（hash 匹配）
- 一段知识不再分裂在 text 列 + images_json 列 + 文件系统三处

### 覆盖学科

所有看结构图/流程图/示意图的学科。

## L3：生成图（选做）

### 目标

LLM 在出题时自己生成结构图（Mermaid/SVG）。

### 适用题型

- 数据结构：让 LLM 生成二叉树/图的 Mermaid 描述，问遍历顺序
- 操作系统：生成状态转换图/调度甘特图
- 计算机网络：生成拓扑图/协议交互时序图

### Mermaid 子集

不用支持全部 Mermaid 语法，只支持：

```
graph TD    → 流程图（状态转换、算法流程）
flowchart   → 同上
gantt       → 调度时序
classDiagram → UML（可选）
```

LLM 生成 Mermaid 源码 → 前端用 mermaid.js 渲染。

### 覆盖学科

数据结构、操作系统、计算机网络、算法。

## L4：交互图（远期）

### 目标

让学生在图上面标注/画/拖。

### 通用协议（先于任何 Canvas 实现）

所有交互式题型共享一个协议，Canvas 组件不存业务逻辑：

```json
{
  "config": {
    "canvas_type": "tree",
    "initial_state": { "nodes": [...], "edges": [...] },
    "tools": ["add_node", "delete_node", "draw_edge"]
  },
  "expected_answer": {
    "nodes": [...], "edges": [...]
  },
  "judge_schema": {
    "mode": "structural_diff",
    "ignore_fields": ["x", "y"]
  }
}
```

**前端**：一个 `InteractiveCanvas` 组件，根据 `config.canvas_type` 选渲染器（tree/circuit/geometry），捕获用户操作输出 `student_answer` JSON。

**后端**：一个 `judgeInteractive` 函数，根据 `judge_schema` 比较 `expected_answer` 和 `student_answer`——通用的结构化 diff，不关心是树还是电路。

**加新课**：只需要配一个 config JSON + 一个轻量渲染器。判题逻辑不重写。

### 典型题型

| 学科 | 题型 | canvas_type | tools |
|---|---|---|---|
| 数据结构 | 给树，画出插入后形态 | tree | add_node, delete_node, draw_edge |
| 电路 | 给电路图，标电流方向 | circuit | annotate_node |
| 几何 | 给图，标注角/边 | geometry | annotate_angle, annotate_edge |

## L5：智能判图（远期）

### 目标

用多模态 LLM（GPT-4V / Claude Vision）直接看图判对错。

### 适用

学生上传手写公式/画图的照片，多模态 LLM 判断"画的对不对"。

### 前提

L2 的图片存储 + L4 的交互数据已经积累了大量"标准图"，多模态 LLM 有参考基线。

---

## 建议执行顺序

1. **L1 公式**（1-2 天）：投入小、收益大、覆盖所有理工科计算题
2. **media 字段落地**（0.5 天）：在 questions JSON 里加 `media` 字段，前端加 renderer 分支，L1 的 LaTeX marks 就用它
3. **L2 图片转文字**（2-3 天）：改入库 pipeline，图片描述合并进 text 列
4. **L2 静态图展示**（1-2 天）：前端加 `<img>` 渲染器
5. **L3 Mermaid**（2-3 天）：加 mermaid renderer，写 prompt 让 LLM 生成 Mermaid
6. **L4 交互协议**（先写 plan，再实现）：不急着做，但做之前先定义通用协议

---

## 不变的东西

- 三个 Agent 架构不变
- BKT + Bandit 算法不变
- Web/API 入口不变
- 数据库 schema 向后兼容（只加列）

---

## 关键原则

- **按学科增量加**：不是做全科目通用，而是遇到一门新课加一种题型
- **降低判题门槛**：图形转文字描述后走 LLM 文字判题，不做视觉判题（L5 之前）
- **不做通用图形引擎**：不同学科用不同的渲染方式（KaTeX / Mermaid / img / Canvas）
- **数据不分裂**：图片用 content-hash 命名，图片描述合并进 text 列，一段知识不散落在多个存储介质里
- **字段不膨胀**：所有富媒体能力统一走 `media` 字段，不加 per-type column
- **协议先于实现**：L4 的交互式题型先定义通用 config/answer/judge_schema 协议，再写组件
