# 学生画像长期记忆闭环计划

## 1. 背景

当前项目已经具备学生画像的核心地基：

- `attempts`：保存学生每次作答事实。
- `attempt_error_labels`：保存错误作答的错因标签。
- `profile_engine.py`：从作答记录实时聚合当前画像。
- `ProfileGraph`：将画像聚合封装成 LangGraph 节点。
- `recommendation.py`：基于 BKT `P(L)` 和 Thompson Sampling 生成练习推荐。
- `practice` 模式：已经能消费画像，把推荐计划反哺到出题策略。

但当前系统主要回答的是：

> 学生现在怎么样？

它还不能稳定回答：

> 学生最近怎么变化？  
> 哪一次练习带来了提升？  
> 哪些弱点是长期存在的？  
> 推荐策略是否真的有效？  
> 下一次练习应该继续当前策略，还是换题型/换难度？

因此，本计划目标是在现有实时画像之上，补齐 `Session → Snapshot → Trend → Memory Facts → Recommendation` 的长期记忆闭环。

## 2. 目标与边界

### 2.1 目标

1. 保留 `attempts` 作为唯一事实源，不用 LLM 总结替代事实。
2. 显式记录一次练习、摸底或考试的边界，即 session。
3. 在关键时间点保存画像快照，支持练前/练后对比。
4. 从快照中分析趋势，识别提升、下降、卡住、遗忘风险。
5. 将稳定趋势沉淀为长期记忆事实，供画像页、学习报告、practice 推荐使用。
6. 让 practice 推荐不只看当前弱点，也能参考长期趋势和推荐效果。

### 2.2 非目标

1. 不重写现有判题链路。
2. 不推翻现有 BKT + Bandit 推荐引擎。
3. 不在第一阶段引入向量库。
4. 不让 LLM 直接维护画像真相。
5. 不把 diagnostic / exam 当作和 practice 等价的训练收益来源。

## 3. 总体架构

长期记忆闭环分为 6 层：

```text
第 1 层：事实层
第 2 层：Session 层
第 3 层：当前画像层
第 4 层：Snapshot / Trend 层
第 5 层：长期记忆事实层
第 6 层：应用层
```

### 3.1 第 1 层：事实层

现有表：

```text
attempts
attempt_error_labels
```

职责：

- 记录学生做了哪道题。
- 记录对错、题型、难度、章节、知识点。
- 记录耗时、信心、判题理由。
- 记录错因诊断。

原则：

- 这是所有画像、趋势、记忆的唯一事实来源。
- 后续新增表只能引用或聚合它，不能替代它。

### 3.2 第 2 层：Session 层

新增表：

```text
learning_sessions
attempts.session_id
```

职责：

- 把零散作答归入一次明确的学习活动。
- 记录模式：`practice` / `diagnostic` / `exam`。
- 记录练习开始、结束、推荐计划、实际题目、结果摘要。
- 通过 `attempts.session_id` 建立 session 与作答记录的关系。

回答的问题：

- 这是哪一次练习？
- 什么时候开始和结束？
- 它是 practice、diagnostic 还是 exam？
- 这次系统推荐了什么？
- 学生实际做了哪些题？
- 这一轮本身的正确率是多少？

设计决策：

- 一次作答天然只属于一次 session，因此不使用 `session_attempts` 多对多关联表。
- 存量 attempt 的 `session_id` 可以为 `NULL`，后续可通过历史回填生成 `mode='historical'` 的合成 session。

### 3.3 第 3 层：当前画像层

现有模块：

```text
exam/student_profile/profile_engine.py
exam/graph/profile_graph.py
```

职责：

- 从 `attempts + attempt_error_labels` 实时聚合当前画像。
- 计算知识点正确率、近期正确率、连续错误、主要错因。
- 使用 BKT 计算每个知识点当前掌握概率 `P(L)`。
- 输出当前弱点、掌握概况、风险信号。

回答的问题：

- 学生现在整体正确率是多少？
- 每个知识点当前掌握概率是多少？
- 当前最薄弱的知识点是什么？
- 当前主要错因是什么？

### 3.4 第 4 层：Snapshot / Trend 层

新增表和模块：

```text
profile_snapshots
exam/student_profile/trend_engine.py
```

职责：

- 在练习前后、摸底后、考试后保存画像快照。
- 对比快照，计算趋势。
- 识别提升、下降、长期卡住、久未复习、推荐无效等信号。

回答的问题：

- 这轮练习前后 `P(L)` 变化多少？
- 哪些知识点最近持续上升？
- 哪些知识点最近持续下降？
- 哪些知识点练了很多但没有提升？
- 哪些知识点太久没复习？

### 3.5 第 5 层：长期记忆事实层

新增表和模块：

```text
student_memory_facts
exam/student_profile/memory_engine.py
```

职责：

- 将趋势沉淀为稳定、可解释、可更新的长期事实。
- 每条记忆都带证据、置信度和生命周期。
- 给推荐、画像页、学习报告提供长期上下文。

示例：

```text
weak_topic: 2.3 任务调度长期薄弱
error_pattern: 综合题经常出现迁移失败
risk: 高信心错误偏多，可能存在伪掌握
trend: 2.1 进程状态转换最近明显提升
strategy_effect: 简答题训练对 2.3 任务调度提升明显
```

回答的问题：

- 这个学生长期的学习特点是什么？
- 哪些弱点不是偶然错误，而是稳定问题？
- 哪些训练策略对这个学生有效？

### 3.6 第 6 层：应用层

涉及入口：

```text
web/server.py
web/index.html
exam/graph/strategy.py
exam/student_profile/recommendation.py
show_profile.py
```

职责：

- 在画像页展示当前画像、趋势和长期记忆。
- 在 practice 出题时使用长期趋势和记忆事实。
- 在学习报告中解释为什么推荐这些题。
- 为后续周报、考前复习计划、教师视图预留数据。

## 4. 模式对接规则

三种出题模式都应该接入 session 和 snapshot，但权重不同。

### 4.1 practice 模式

定位：

```text
训练模式，完整闭环。
```

对接：

1. 出题前创建 session。
2. 出题前保存 `pre_snapshot`。
3. 保存本轮推荐计划。
4. 交卷后写入 attempts。
5. 交卷后保存 `post_snapshot`。
6. 计算练前/练后 `delta_mastery`。
7. 更新 session 结果。
8. 更新趋势和长期记忆。
9. 将 reward 反馈给后续 Bandit 推荐。

是否计入训练收益：

```text
是，高权重。
```

### 4.2 diagnostic 模式

定位：

```text
摸底模式，用于校准初始画像。
```

对接：

1. 创建 session。
2. 交卷后保存 snapshot。
3. 更新当前画像。
4. 可作为初始弱点识别依据。
5. 不直接计算 Bandit reward。

是否计入训练收益：

```text
否，或极低权重。
```

原因：

- diagnostic 的题目通常是覆盖式、低难度、探测性质。
- 它适合判断“起点”，不适合判断“练习策略是否有效”。

### 4.3 exam 模式

定位：

```text
评估模式，用于反映综合水平。
```

对接：

1. 创建 session。
2. 交卷后保存 snapshot。
3. 错题和错因进入事实层。
4. 可低权重影响长期记忆。
5. 不作为普通 practice reward 使用。

是否计入训练收益：

```text
否，或低权重。
```

原因：

- exam 更接近评估，不是系统推荐后执行的训练。
- 它能暴露风险，但不应被当作“某个推荐策略有效/无效”的直接证据。

## 5. 数据表设计草案

### 5.1 `learning_sessions`

用途：记录一次学习活动。

```sql
CREATE TABLE IF NOT EXISTS learning_sessions (
    id INTEGER PRIMARY KEY,
    student_id TEXT NOT NULL,
    mode TEXT NOT NULL,                 -- practice / diagnostic / exam
    status TEXT DEFAULT 'active',        -- active / completed / aborted

    started_at TEXT DEFAULT (datetime('now')),
    ended_at TEXT,

    pre_snapshot_id INTEGER,
    post_snapshot_id INTEGER,

    recommendation_json TEXT DEFAULT '',
    focus_sections_json TEXT DEFAULT '',
    focus_topics_json TEXT DEFAULT '',
    question_types_json TEXT DEFAULT '',
    target_count INTEGER DEFAULT 0,

    attempt_count INTEGER DEFAULT 0,
    correct_count INTEGER DEFAULT 0,
    accuracy REAL DEFAULT 0.0,

    delta_mastery_json TEXT DEFAULT '',
    effect_summary TEXT DEFAULT '',

    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);
```

关键说明：

- `recommendation_json` 保存出题前生成的推荐计划。
- `pre_snapshot_id` 和 `post_snapshot_id` 连接快照。
- `delta_mastery_json` 保存本轮每个知识点的 `P(L)` 变化。
- `mode` 决定后续趋势和记忆更新的权重。
- `status` 保持三态：`active` 表示进行中，`completed` 表示交卷、写入 attempts、保存 post snapshot 均完成，`aborted` 表示用户中止或服务端放弃。

### 5.2 `attempts.session_id`

用途：直接把作答记录归属到 session。

```sql
ALTER TABLE attempts ADD COLUMN session_id INTEGER;
```

说明：

- 这是 nullable 字段，存量历史数据可以保持 `NULL`。
- 新作答写入时直接带上当前 `session_id`。
- 查询某次 session 的作答只需要 `WHERE session_id = ?`，不需要额外 JOIN。
- 如果后续做历史回填，可按学生和时间窗口创建 `mode='historical'` 的合成 session，并把旧 attempts 填入对应 `session_id`。

### 5.3 `profile_snapshots`

用途：保存某一时刻的画像结果。

```sql
CREATE TABLE IF NOT EXISTS profile_snapshots (
    id INTEGER PRIMARY KEY,
    student_id TEXT NOT NULL,
    session_id INTEGER,
    snapshot_type TEXT NOT NULL,         -- pre / post / manual
    profile_version TEXT NOT NULL,       -- 如 bkt-v1-threshold-v1

    total_attempts INTEGER DEFAULT 0,
    overall_accuracy REAL DEFAULT 0.0,

    profile_json TEXT NOT NULL,

    created_at TEXT DEFAULT (datetime('now'))
);
```

关键说明：

- `snapshot_type` 只表示快照时间点，不把模式揉进枚举里。
- `mode` 从 `learning_sessions.mode` 通过 `session_id` 关联得到。
- `profile_json` 直接保存 `build_profile()` 的结构化输出，避免过早拆分多个 JSON 列。
- `profile_version` 必须保存画像算法版本，防止 BKT 参数或聚合规则变更后跨版本快照不可比。
- `total_attempts` 和 `overall_accuracy` 是列表展示用的少量冗余标量，不作为画像真相源。
- 快照是派生数据，可以重建，但保存后能支持趋势、报告和效果追踪。

### 5.4 `student_memory_facts`

用途：保存长期稳定结论。

```sql
CREATE TABLE IF NOT EXISTS student_memory_facts (
    id INTEGER PRIMARY KEY,
    student_id TEXT NOT NULL,

    memory_type TEXT NOT NULL,            -- weak_topic / error_pattern / trend / risk / strategy_effect / preference
    memory_key TEXT NOT NULL,             -- 如 section_id、error_type、question_type
    value_json TEXT DEFAULT '',

    confidence REAL DEFAULT 0.0,
    evidence_json TEXT DEFAULT '',

    first_seen TEXT DEFAULT (datetime('now')),
    last_seen TEXT DEFAULT (datetime('now')),
    status TEXT DEFAULT 'active',         -- active / stale / resolved / dismissed

    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now')),

    UNIQUE(student_id, memory_type, memory_key)
);
```

关键说明：

- `confidence` 根据证据次数、持续时间、趋势稳定性更新。
- `status=resolved` 可表示长期弱点已经解除。
- `status=stale` 可表示很久没有新证据，推荐时降权。

## 6. 新增模块规划

### 6.1 `exam/student_profile/session_storage.py`

职责：

- 初始化 session / snapshot / memory 表。
- 创建 session。
- 更新 session。
- 将当前 session_id 写入 attempts。
- 保存 snapshot。
- 查询最近 sessions。
- 查询和更新长期记忆事实。

边界：

- `exam/student_profile/storage.py` 继续负责 `attempts` 与 `attempt_error_labels` 的事实层写入。
- `session_storage.py` 负责 `learning_sessions`、`profile_snapshots`、`student_memory_facts`。
- 跨层编排放在 `web/server.py` 或专门 service 层，不让两个 storage 文件互相调用。
- 暂不重命名 `storage.py`，避免引入无关 import 迁移；后续如文件继续膨胀，再考虑拆为 `attempt_storage.py`。

建议函数：

```python
init_long_memory_db(db_path: str) -> None
create_learning_session(db_path: str, student_id: str, mode: str, ...) -> int
finish_learning_session(db_path: str, session_id: int, summary: dict) -> None
attach_attempts_to_session(db_path: str, session_id: int, attempt_ids: list[int]) -> None
save_profile_snapshot(db_path: str, student_id: str, profile: StudentProfile, profile_version: str, ...) -> int
get_recent_sessions(db_path: str, student_id: str, limit: int = 5) -> list[dict]
```

### 6.2 `exam/student_profile/trend_engine.py`

职责：

- 对比两个 snapshot。
- 分析最近 N 个 snapshot。
- 输出趋势摘要。

建议函数：

```python
compare_snapshots(pre: dict, post: dict) -> dict
build_trend_summary(db_path: str, student_id: str, window: int = 5) -> dict
detect_stalled_topics(trends: dict) -> list[dict]
detect_declining_topics(trends: dict) -> list[dict]
detect_improving_topics(trends: dict) -> list[dict]
```

趋势分类建议：

```text
improving: 最近多次 P(L) 稳定上升
declining: 最近多次 P(L) 下降
stalled: 多次练习后 P(L) 变化小于阈值
volatile: 大幅波动，不稳定
forgotten_risk: 已掌握但久未复习
```

### 6.3 `exam/student_profile/memory_engine.py`

职责：

- 从趋势摘要中生成或更新长期记忆事实。
- 处理记忆的增强、衰减、解除。
- 为推荐返回可用的长期上下文。

建议函数：

```python
update_memory_facts(db_path: str, student_id: str, trend_summary: dict) -> list[dict]
get_active_memory_facts(db_path: str, student_id: str) -> list[dict]
mark_resolved_if_recovered(db_path: str, student_id: str, topic_state: dict) -> None
decay_stale_memories(db_path: str, student_id: str) -> None
```

记忆更新原则：

- 连续多次出现才提升置信度。
- 单次异常不生成强记忆。
- 新证据可以覆盖旧结论。
- 记忆必须可追溯到 snapshot 或 session。
- LLM 只负责表达，不负责决定事实是否成立。

## 7. API 对接设计

### 7.1 `/api/generate`

当前职责：

```text
生成题目，并更新服务器内存中的 QUESTIONS。
```

新增行为：

当 `mode=practice`：

1. 创建 `learning_session(status='active')`。
2. 调用 `build_profile(student_id, mastery_backend="bkt")`。
3. 保存 `pre_snapshot(snapshot_type='pre')`，并回写 `learning_sessions.pre_snapshot_id`。
4. 运行出题流程，拿到本轮 focus、题型、题数和 practice plan。
5. 将推荐计划和出题范围写入 session。
6. 返回 `session_id` 给前端。

当 `mode=diagnostic` 或 `mode=exam`：

1. 创建 `learning_session`。
2. 可不保存 pre snapshot，或只在需要对比时保存。
3. 返回 `session_id`。

响应建议增加：

```json
{
  "ok": true,
  "count": 20,
  "mode": "practice",
  "session_id": 123
}
```

### 7.2 `/api/submit-exam`

当前职责：

```text
判题 -> 写 attempts -> 返回结果。
```

新增行为：

1. 接收前端传回的 `session_id`。
2. 判题并写入 attempts。
3. 写入 attempts 时带上 `session_id`，并返回本次新增的 attempt ids。
4. 构建 post profile。
5. 保存 `post_snapshot(snapshot_type='post')`。
6. 对 practice 模式计算 `delta_mastery`。
7. 更新 `learning_sessions(status='completed')`。
8. 对 practice 模式触发 trend / memory 更新。

实现要求：

- `record_attempts_batch()` 需要支持写入 `session_id`，并返回本轮新增 attempt ids。
- 如果前端没有传 `session_id`，允许降级写入 attempts，但该轮不进入精确 session reward；可在日志中标记为 orphan attempts。

请求建议增加：

```json
{
  "student_id": "S001",
  "session_id": 123,
  "answers": []
}
```

响应可选增加：

```json
{
  "ok": true,
  "results": [],
  "session": {
    "id": 123,
    "accuracy": 0.75,
    "effect_summary": "2.3 任务调度 P(L) +11%"
  }
}
```

### 7.3 `/api/profile`

当前职责：

```text
返回当前画像、推荐计划、错因分布、风险信号。
```

新增返回：

```json
{
  "recent_sessions": [],
  "trend_summary": {},
  "memory_facts": []
}
```

前端展示建议：

- 当前画像：沿用现有 UI。
- 最近练习：展示最近 3-5 个 session。
- 趋势：展示重点知识点 `P(L)` 变化。
- 长期记忆：展示稳定弱点、主要错因、策略效果。

## 8. 前端对接设计

涉及文件：

```text
web/index.html

```

### 8.1 出题页

新增状态：

```javascript
let CURRENT_SESSION_ID = null;
```

```javascript
CURRENT_SESSION_ID = data.session_id;
```

### 8.2 画像页

新增区域：

1. 最近练习记录。
2. 掌握度变化。
3. 长期记忆事实。

展示原则：

- 不堆过多指标。
- 先给结论，再给证据。
- 能解释推荐原因。

示例：

```text
最近变化：
- 2.3 任务调度：P(L) 32% → 43%，本轮练习有效
- 3.1 临界区：P(L) 48% → 44%，仍不稳定

长期记忆：
- 任务调度长期薄弱，主要错因为概念混淆
- 综合题迁移失败出现 4 次，建议增加综合题训练
```

## 9. 推荐引擎升级设计

当前推荐依据：

```text
BKT P(L)
+ dominant_error_type
+ Thompson Sampling
+ session_rewards
```

升级后新增依据：

```text
trend_summary
+ memory_facts
+ recent_session_effect
+ forgotten_risk
```

### 9.1 推荐权重建议

practice 模式推荐 score 可逐步演化为：

```text
score =
  current_weakness_score
+ long_term_weakness_bonus
+ declining_trend_bonus
+ stalled_topic_bonus
+ previous_strategy_effect_bonus
+ forgotten_risk_bonus
- mastered_recently_penalty
```

第一阶段不必完整实现该公式。Phase 4 再为 `build_recommendation_plan()` 增加可选参数，例如 `trend_summary=None`、`memory_facts=None`，避免在 MVP 阶段提前改推荐接口。

### 9.2 策略分支

```text
当前弱 + 长期弱 + 上次练有效
→ 继续巩固，同类题型稍微提高难度

当前弱 + 长期弱 + 多次练无效
→ 换题型，降低难度，优先概念澄清

当前不弱 + 最近下降
→ 少量复习，防止遗忘

当前掌握 + 久未复习
→ 插入 1 道回顾题

高信心错误偏多
→ 增加易混选项和解释型题目
```

### 9.3 `compute_session_rewards()` 迁移路线

现状：

```text
profile_engine.py 中的 compute_session_rewards()
→ 按 30 分钟间隔硬切 session
→ 为 Bandit 提供 session reward
```

引入显式 session 后，reward 来源必须收敛，否则会出现两套 session 定义。

迁移策略：

1. Phase 1a 先保留旧函数，确保现有 practice 推荐不受影响。
2. Phase 1c 增加历史回填脚本：把 `session_id IS NULL` 的历史 attempts 按学生和 30 分钟间隔合成为 `mode='historical'` session。
3. Phase 2 新增 `compute_explicit_session_rewards()`，只从 `learning_sessions + attempts.session_id + profile_snapshots` 计算 reward。
4. Phase 2 末尾将 practice 推荐切到显式 session reward。
5. 旧 `compute_session_rewards()` 只保留为历史回填工具或兼容降级路径，并在注释中标记不再作为主推荐 reward 来源。

历史兼容原则：

- 没有 `session_id` 的旧数据不丢弃。
- 回填只做一次，保留 `mode='historical'`，不伪装成真实 practice。
- diagnostic / exam 的 session 可以进入画像事实层，但不作为高权重训练 reward。

## 10. 实施阶段

### Phase 0：现状保护

目标：

- 不改变现有画像输出。
- 不改变现有判题行为。
- 不改变现有 practice 推荐结果。

任务：

- 阅读并确认 `storage.py`、`profile_engine.py`、`recommendation.py`、`web/server.py` 当前行为。
- 确认 `attempts.db` 初始化位置。
- 明确新增表应与 `attempts.db` 共库存储。

验收：

- 原有 `show_profile.py` 可继续使用。
- 原有网页答题流程不受影响。

### Phase 1a：后端 Session + Snapshot 闭环

目标：

- 后端能记录一次学习活动。
- practice 能保存练前/练后画像快照。
- attempts 能直接归属到 session。

任务：

1. 新增 `session_storage.py`。
2. 初始化 `learning_sessions`、`profile_snapshots`、`student_memory_facts`。
3. 对 `attempts` 增加 nullable `session_id` 字段。
4. `/api/generate` 在 practice 模式创建 active session，并保存 pre snapshot。
5. `record_attempts_batch()` 支持写入 `session_id` 并返回 attempt ids。
6. `/api/submit-exam` 保存 post snapshot，并将 session 更新为 completed。
7. diagnostic / exam 模式创建 session，并在交卷后保存 post snapshot。

验收：

- practice 出题后数据库出现一条 `status='active'` 的 session。
- practice 交卷后 session 状态变为 `completed`。
- 本轮新增 attempts 的 `session_id` 等于当前 session。
- practice 有 pre 和 post 两个 snapshot。
- diagnostic / exam 至少有 post snapshot。
- 不需要前端展示也能通过 API 响应和数据库内容验证。

### Phase 1b：前端 Session 透传与画像页展示

目标：

- 前端能保存并传回 `session_id`。
- 画像页能展示最近 session。

任务：

1. `/api/generate` 响应中返回 `session_id`。
2. `web/index.html` 保存 `CURRENT_SESSION_ID`。
3. `/api/submit-exam` 请求中带上 `session_id`。
4. `/api/profile` 返回 `recent_sessions`。
5. 画像页展示最近 3-5 次 session 摘要。

验收：

- 用户通过网页 practice 出题后，交卷请求包含同一个 `session_id`。
- 画像页能看到最近练习记录、模式、题数、正确率和完成时间。

### Phase 1c：历史数据回填

目标：

- 让存量 attempts 也能进入新 session 体系。
- 为后续显式 session reward 迁移铺路。

任务：

1. 新增一次性回填脚本或管理函数。
2. 按 `student_id` 和 30 分钟间隔切分 `session_id IS NULL` 的历史 attempts。
3. 创建 `mode='historical'`、`status='completed'` 的合成 session。
4. 将旧 attempts 的 `session_id` 回填到对应 historical session。
5. 为 historical session 保存 post snapshot，必要时不生成 pre snapshot。

验收：

- 回填后旧 attempts 不再全部是 `session_id IS NULL`。
- historical session 不参与高权重 practice reward。
- 回填逻辑可重复执行且不会重复创建 session。

### Phase 2：Delta Mastery 与趋势分析

目标：

- 能回答每次练习带来的掌握度变化。
- 能识别最近提升、下降、卡住的知识点。
- 将推荐 reward 从隐式时间切分迁移到显式 session。

任务：

1. 新增 `trend_engine.py`。
2. 实现 `compare_snapshots()`。
3. 在 session 中写入 `delta_mastery_json`。
4. 实现 `build_trend_summary()`。
5. 实现 `compute_explicit_session_rewards()`。
6. 将 practice 推荐的 reward 来源切换到显式 session。
7. `/api/profile` 返回 `trend_summary`。
8. 前端展示最近变化。

验收：

- practice 结束后能看到每个推荐知识点的 `P(L)` 变化。
- 最近趋势能区分 improving / declining / stalled。
- practice 推荐不再依赖 30 分钟硬切 session 作为主 reward 来源。
- 趋势结果不影响当前画像计算。

### Phase 3：长期记忆事实

目标：

- 从趋势中沉淀稳定结论。
- 支持记忆增强、降权、解除。

任务：

1. 新增 `memory_engine.py`。
2. 初始化 `student_memory_facts`。
3. 从 trend summary 生成 memory facts。
4. 实现 active / stale / resolved 状态更新。
5. `/api/profile` 返回 `memory_facts`。
6. 前端展示长期记忆。

验收：

- 连续多次薄弱的知识点会生成 `weak_topic` 记忆。
- 持续改善后相关记忆可以降权或 resolved。
- 每条记忆有 evidence 和 confidence。

### Phase 4：推荐策略接入长期记忆

目标：

- practice 推荐开始利用趋势和长期记忆。

任务：

1. `strategy_router()` 读取 trend summary 和 memory facts。
2. `build_recommendation_plan()` 支持可选长期上下文。
3. 对长期薄弱、下降、卡住、久未复习增加排序权重。
4. 对多次无效的练习策略切换题型或难度。
5. 在 `practice_plan.reason` 中解释长期依据。

验收：

- 长期薄弱点在 practice 中被稳定优先考虑。
- 已掌握但久未复习的点可被少量插入。
- 多次练习无效的点不再简单重复同一种题型。

### Phase 5：学习报告与可视化

目标：

- 将长期记忆价值体现在产品体验和毕设展示中。

任务：

1. 画像页增加趋势图。
2. 增加最近练习效果摘要。
3. 增加长期弱点列表。
4. 增加考前风险清单。
5. 可选：导出学习报告。

验收：

- 用户能看懂自己哪里变好了。
- 用户能看懂为什么系统推荐下一轮练习。
- 系统能展示推荐闭环效果。

## 11. 风险与处理

### 11.1 画像快照 JSON 过大

风险：

- `profile_json` 如果完整保存所有画像字段，可能随数据增长变大。

处理：

- `profile_json` 只保存画像和趋势分析必要字段。
- 大文本题干、答案不进入 snapshot。
- `profile_version` 必填，算法变化后按版本区分可比性。

### 11.2 session 与 attempts 关联不准

风险：

- 如果交卷时没有 `session_id`，无法知道 attempts 属于哪轮练习。

处理：

- 前端必须传 `session_id`。
- 后端允许 fallback：作答照常写入，但 `session_id` 为空的 orphan attempts 不进入精确 session reward。
- 历史回填脚本可以按 student_id 和时间窗口处理旧数据，但线上交卷不依赖时间窗口做主路径关联。

### 11.3 diagnostic / exam 污染训练 reward

风险：

- 把摸底或考试当成训练收益，会误导 Bandit。

处理：

- session 必须有 `mode`。
- reward 只对 practice 高权重生效。

### 11.4 长期记忆误判

风险：

- 单次异常生成长期事实，会影响推荐。

处理：

- 记忆生成需要多次证据。
- 每条记忆带 confidence。
- 低置信度记忆只展示，不强影响推荐。

### 11.5 LLM 总结污染事实

风险：

- LLM 生成听起来合理但无证据的结论。

处理：

- LLM 只做表达层。
- `student_memory_facts` 的生成依据必须来自 trend / snapshot / attempts。

## 13. 最小可行版本

最小版本只做两件事：

```text
learning_sessions
+ profile_snapshots
+ attempts.session_id
```

具体范围：

1. practice 出题时创建 session。
2. practice 出题前保存 pre snapshot。
3. 交卷时把 `session_id` 写入 attempts。
4. practice 交卷后保存 post snapshot。
5. 计算 `delta_mastery_json`。
6. `/api/profile` 返回最近 session 和本轮变化。

MVP 可以先只做后端闭环。前端展示放在 Phase 1b，不阻塞数据库和 API 正确性验证。

这一步完成后，系统就能从：

```text
现在哪里弱
```

升级为：

```text
哪次练习让哪里变好了
```

后续长期记忆事实和推荐升级都可以自然接上。

## 14. 推荐落地顺序

建议按以下顺序实施：

1. 新增 `session_storage.py`，初始化 `learning_sessions`、`profile_snapshots`、`student_memory_facts`。
2. 给 `attempts` 增加 nullable `session_id`。
3. 修改 `/api/generate`，让 practice 创建 session 和 pre snapshot。
4. 修改 `/api/submit-exam`，写入 `session_id`，保存 post snapshot，完成 session。
5. 修改前端，保存并提交 `session_id`。
6. 修改 `/api/profile`，返回 recent sessions。
7. 增加历史回填脚本，生成 `mode='historical'` session。
8. 新增 `trend_engine.py`，计算 delta、趋势和显式 session reward。
9. 新增 `memory_engine.py`，生成长期记忆事实。
10. 最后改 `strategy_router()` 和 `recommendation.py`，让长期记忆反哺 practice 推荐。

## 16. 一句话总结

当前系统已经有“长期作答日志”和“实时画像”。

本计划要补的是：

```text
一次练习的边界
+ 画像历史快照
+ 练习效果趋势
+ 可解释长期记忆
+ 反哺下一次推荐
```

最终目标不是多存几张表，而是让系统从“当前错题推荐”升级为“知道学生长期如何变化的学习教练”。
