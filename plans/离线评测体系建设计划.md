# 离线评测体系建设计划

## 背景

当前系统已经具备教材解析、Agent 出题、混合判题、错因诊断、学生画像和 BKT + Bandit 推荐闭环。但这些能力主要依赖人工体验判断，缺少一套可重复运行的量化评测体系来证明：

- 题目质量是否稳定
- LLM 判题是否可靠
- 推荐策略是否真的提升薄弱知识点掌握度
- Prompt、模型、策略改动后是否发生回退

这会影响项目在简历和面试中的说服力：系统看起来功能完整，但缺少“生产级 Agent 工程”的验证闭环。

## 目标

搭建一套可离线运行、可复现、可扩展的 Agent 评测体系，用固定样本和结构化指标评估三条核心链路：

1. **出题质量评测**：题目格式、答案唯一性、教材依据、难度/题型遵循、重复率。
2. **判题质量评测**：规则判题准确率、LLM 语义判题一致率、错因诊断可用率。
3. **推荐效果评测**：BKT 预测合理性、Bandit 推荐命中薄弱点、练习后 P(L) 提升。

最终产物包括：

- 固定评测样本集
- 离线评测脚本
- 指标 JSON/Markdown 报告
- 可选 Web 评测报告页
- 可写进简历的量化指标

## 核心原则

- **先可复现，再追求复杂**：第一版固定 20 个章节样本和少量人工标注答案即可。
- **先离线，不接入主流程**：评测体系初期不影响用户正常出题/答题。
- **指标分层**：格式类指标自动判断，语义类指标允许人工 golden set 或 LLM-as-judge 辅助。
- **保留原始证据**：每条失败样本必须保存题干、答案、来源章节、错误原因，方便回溯。
- **能做回归**：Prompt、模型或策略改动后，可以重复跑同一批样本对比指标变化。

## 评测对象

### 1. ExamGraph 出题链路

覆盖节点：

- `strategy_router`
- `chief_editor`
- `knowledge_extractor`
- 五类题目生成器
- `quality_reviewer`
- `final_editor`

评估重点：

- 是否按指定章节出题
- 是否按指定题型/难度出题
- 题干是否完整
- 正确答案是否存在且唯一
- 解析是否能支撑答案
- 是否能追溯到教材章节
- 同一批题之间是否高度重复

### 2. JudgeGraph 判题链路

覆盖对象：

- 选择题规则判题
- 填空题规则判题
- 简答题/综合题/代码填空 LLM 判题
- 错因诊断
- LLM 异常降级

评估重点：

- 客观题判题准确率
- 主观题与人工标注的一致率
- false positive / false negative
- 错因标签是否落在合法枚举内
- 错因证据和建议是否非空、可读
- LLM 不可用时是否稳定降级

### 3. ProfileGraph + Recommendation 推荐链路

覆盖对象：

- BKT P(L) 回放
- mastery_level 映射
- session reward
- Thompson Sampling / Beta mean 排序
- recommendation plan 题型、难度、题数建议

评估重点：

- 全对序列 P(L) 是否上升
- 全错序列 P(L) 是否下降
- 薄弱 topic 是否排在推荐前列
- 已掌握 topic 是否自然退场
- session 前后 delta 是否正确写入
- 推荐题型是否匹配主要错因

## 评测样本设计

### 样本集目录建议

后续实现时建议新增：

```text
evals/
  cases/
    generation_cases.json
    judge_cases.json
    recommendation_cases.json
  golden/
    judge_golden.json
    topic_grounding.json
  reports/
    eval_YYYYMMDD_HHMMSS.json
    eval_YYYYMMDD_HHMMSS.md
```

### 出题样本 generation_cases.json

第一版固定 20 个章节样本，覆盖：

- 高频考试章节
- 容易混淆章节
- 有代码/算法/流程的章节
- 概念型章节
- 已知出题质量不稳定章节

字段建议：

```json
{
  "case_id": "gen_001",
  "section_id": "2.3",
  "topic_hint": "任务调度",
  "mode": "exam",
  "target_count": 3,
  "allowed_types": ["choice", "fill_blank", "short_answer"],
  "allowed_difficulty": ["easy", "medium"],
  "expected_keywords": ["调度", "优先级", "就绪"]
}
```

### 判题样本 judge_cases.json

每种题型准备正确、错误、部分正确、空答案、边界格式各若干条。

字段建议：

```json
{
  "case_id": "judge_001",
  "question_type": "choice",
  "stem": "以下哪个函数会引起任务调度？",
  "options": ["A. OSTimeDly()", "B. OSInit()", "C. printf()", "D. malloc()"],
  "correct_answer": "A",
  "student_answer": "A",
  "expected_correct": true,
  "expected_error_type": null
}
```

### 推荐样本 recommendation_cases.json

用合成 attempts 构造可控学生画像，不依赖真实用户数据。

典型场景：

- topic A 连续答错，应优先推荐
- topic B 最近提升明显，应继续巩固
- topic C 已掌握，应降低推荐权重
- topic D 样本不足，应保留探索机会

字段建议：

```json
{
  "case_id": "rec_001",
  "student_id": "eval_student_001",
  "attempts": [
    {"section_id": "2.1", "topic": "任务状态", "is_correct": false},
    {"section_id": "2.1", "topic": "任务状态", "is_correct": false},
    {"section_id": "3.2", "topic": "信号量", "is_correct": true}
  ],
  "expected_top_sections": ["2.1"],
  "expected_low_priority_sections": ["3.2"]
}
```

## 指标体系

### 出题质量指标

| 指标 | 含义 | 计算方式 |
|---|---|---|
| format_pass_rate | 格式合格率 | 必填字段完整的题目数 / 总题数 |
| type_adherence_rate | 题型遵循率 | 题型在 allowed_types 内的题目数 / 总题数 |
| difficulty_adherence_rate | 难度遵循率 | 难度在 allowed_difficulty 内的题目数 / 总题数 |
| answer_presence_rate | 答案存在率 | correct_answer 非空题目数 / 总题数 |
| explanation_presence_rate | 解析存在率 | explanation 非空题目数 / 总题数 |
| grounding_hit_rate | 教材依据命中率 | 题干/解析命中章节关键词或来源片段的题目数 / 总题数 |
| duplicate_rate | 重复率 | 相似题数量 / 总题数 |
| quality_retry_rate | 质检重试率 | 质检失败重试次数 / 总生成题数 |

第一版可先实现自动指标：

- `format_pass_rate`
- `type_adherence_rate`
- `difficulty_adherence_rate`
- `answer_presence_rate`
- `explanation_presence_rate`
- `duplicate_rate`

第二版再补：

- `grounding_hit_rate`
- `answer_uniqueness_score`
- `llm_quality_score`

### 判题质量指标

| 指标 | 含义 | 计算方式 |
|---|---|---|
| objective_accuracy | 客观题准确率 | 选择/填空预测正确数 / 客观题总数 |
| subjective_agreement | 主观题一致率 | LLM 判题与人工 golden 一致数 / 主观题总数 |
| false_positive_rate | 错判为对比例 | 实际错误但判对数量 / 实际错误数量 |
| false_negative_rate | 对判为错比例 | 实际正确但判错数量 / 实际正确数量 |
| error_type_valid_rate | 错因合法率 | 错因在枚举内的数量 / 有错因样本数 |
| diagnosis_completeness | 诊断完整率 | evidence/suggestion 非空数量 / 错题数量 |
| fallback_pass_rate | 降级稳定率 | 模拟 LLM 不可用时无异常返回的样本数 / 总样本数 |

### 推荐效果指标

| 指标 | 含义 | 计算方式 |
|---|---|---|
| bkt_monotonic_pass | BKT 单调性 | 全对上升、全错下降的 case 是否通过 |
| weak_topic_hit_rate | 薄弱点命中率 | expected_top_sections 出现在推荐 TopK 的比例 |
| mastered_retire_rate | 已掌握退场率 | mastered topic 未进入 TopK 的比例 |
| recommendation_reason_rate | 推荐原因完整率 | reason_text 非空推荐项 / 推荐项总数 |
| error_to_type_match_rate | 错因题型匹配率 | 推荐题型匹配 ERROR_TYPE_QUESTIONS 的比例 |
| delta_mastery_valid_rate | 掌握度增量有效率 | session delta 可计算且范围合理的比例 |

## 实施方案

### Phase 0 — 基线定义

目标：先固定评测口径，避免后面指标漂移。

任务：

- 确定第一批 20 个章节样本
- 确定题型覆盖：choice、fill_blank、short_answer、code_fill、comprehensive
- 确定输出报告字段
- 明确第一版不做复杂 LLM-as-judge，只做自动指标 + 少量人工 golden

产物：

- `evals/cases/generation_cases.json`
- `evals/cases/judge_cases.json`
- `evals/cases/recommendation_cases.json`
- `evals/README.md`

### Phase 1 — 出题质量评测 MVP

目标：能对固定章节批量出题，并输出基础质量指标。

建议新增：

- `exam/evaluation/__init__.py`
- `exam/evaluation/schemas.py`
- `exam/evaluation/generation_eval.py`
- `exam/evaluation/report.py`
- `scripts/run_generation_eval.py`

核心流程：

```text
读取 generation_cases
  -> 调 ExamGraph 或单题生成链路
  -> 收集 questions
  -> 计算格式/题型/难度/答案/解析/重复率
  -> 输出 JSON + Markdown 报告
```

第一版不要强行全链路跑很多题，避免 LLM 成本过高。可以先支持：

- `--limit 5`
- `--case gen_001`
- `--use-latest-output` 直接评测 `output/questions_*.json`

验收标准：

- 能从最新 `output/questions_*.json` 生成评测报告
- 报告中至少包含 6 个自动指标
- 每个失败项能列出题号和原因

### Phase 2 — 判题一致性评测

目标：验证 JudgeGraph 的可靠性。

建议新增：

- `exam/evaluation/judge_eval.py`
- `scripts/run_judge_eval.py`

核心流程：

```text
读取 judge_cases
  -> 调 JudgeGraph
  -> 对比 expected_correct / expected_error_type
  -> 统计准确率、误判率、错因合法率
  -> 模拟 llm_client=None 验证降级路径
```

验收标准：

- 客观题准确率应接近 100%
- LLM 不配置时不崩溃
- 错因字段必须落在 `ERROR_TYPES` 内
- 报告列出 false positive / false negative 明细

### Phase 3 — 推荐策略离线回放

目标：证明 BKT + Bandit 推荐不是装饰，而是能稳定命中薄弱点。

建议新增：

- `exam/evaluation/recommendation_eval.py`
- `scripts/run_recommendation_eval.py`

核心流程：

```text
创建临时 attempts.db
  -> 写入合成 attempts
  -> build_profile(mastery_backend="bkt")
  -> build_recommendation_plan(rank_strategy="mean")
  -> 对比 expected_top_sections / expected_low_priority_sections
  -> 输出推荐命中率、退场率、原因完整率
```

注意：

- 使用临时数据库，不能污染 `cache/attempts.db`
- 排序使用 `rank_strategy="mean"`，避免 Thompson Sampling 随机性影响评测稳定性
- 对随机策略的评测放到后续 Monte Carlo 版本

验收标准：

- 可稳定复现同一推荐排序
- 弱点命中 Top3
- 已掌握 topic 不进入 Top3
- 推荐原因、题型、难度字段完整

### Phase 4 — 汇总报告与趋势对比

目标：把三类评测汇总成一次完整 Agent Eval 报告。

建议新增：

- `scripts/run_all_evals.py`
- `evals/reports/index.json`
- 可选：`web` 增加评测报告查看 API 和页面

报告结构：

```text
# Agent 离线评测报告

## 总览
- 总分
- 出题质量
- 判题质量
- 推荐质量

## 出题质量
- 指标表
- 失败题目明细

## 判题质量
- 混淆矩阵
- false positive / false negative
- 错因诊断明细

## 推荐质量
- TopK 命中
- BKT P(L) 变化
- 推荐原因覆盖

## 回归对比
- 相比上一次运行提升/下降的指标
```

验收标准：

- 一条命令生成完整报告
- 报告可用于面试展示
- 能和上一次评测结果做 diff

## 数据结构建议

### EvalResult

```python
@dataclass
class EvalMetric:
    name: str
    value: float
    threshold: float | None = None
    passed: bool = True
    detail: str = ""

@dataclass
class EvalFailure:
    case_id: str
    item_id: str
    reason: str

@dataclass
class EvalReport:
    eval_type: str
    run_id: str

```

## 简历包装点

实现后可以写：

> 搭建 Agent 离线评测体系，覆盖出题质量、判题一致性和推荐效果三类指标；通过固定评测集量化题目格式合格率、教材依据命中率、判题准确率和 BKT 推荐命中率，支持 Prompt/模型/策略改动后的自动回归验证。

更工程化版本：

> 设计可复现的 Agent Eval Pipeline，将 LLM 出题、语义判题、BKT + Bandit 推荐拆分为独立评测任务，输出 JSON/Markdown 报告和失败样本证据链，为多 Agent 系统提供质量基线与回归检测能力。

## 不做范围

第一版不做：

- 不接入 CI
- 不引入复杂在线 A/B 测试
- 不做大规模人工标注平台
- 不强依赖外部 LangSmith 服务
- 不改变现有 Web 出题/答题流程

后续可做：

- 接 LangSmith / OpenTelemetry 做 trace 对齐
- 引入 LLM-as-judge 评估题目质量
- 加 embedding 相似度去重
- 加评测报告 Web 页面
- 加模型对比：DeepSeek vs OpenAI vs 本地模型

## 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| LLM 输出随机导致指标波动 | 回归结果不稳定 | 固定 temperature、限制样本数、保留多次运行均值 |
| 评测成本过高 | 不愿频繁运行 | 支持 `--use-latest-output` 和 `--limit` |
| LLM-as-judge 本身不可靠 | 质量分数失真 | 第一版以规则指标和人工 golden 为主 |
| 推荐评测受随机采样影响 | 排序不稳定 | 离线评测使用 Beta mean，不用 Thompson 随机采样 |
| 真实用户数据污染 | 画像结果不可控 | 推荐评测使用临时 SQLite 数据库 |

## 推荐落地顺序

1. 先做 `output/questions_*.json` 的出题质量评测，不调用 LLM，成本最低。
2. 再做 JudgeGraph golden cases，验证判题准确率。
3. 再做推荐临时库回放，证明 BKT + Bandit 有效。
4. 最后做统一报告和趋势对比。

这样每一步都能单独写进简历，不需要等整套体系完成。
