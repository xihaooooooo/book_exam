# Red-Blue Design Review Skill

## 触发方式

用户说 "red-blue review" / "红蓝对抗" / "红蓝审查" / "攻防审查" 时启用。

## 参数

用户需要提供：
- **目标描述**：要审查的系统/功能是什么，当前架构如何
- **审查重点**（可选）：特别关注哪些维度

## 执行流程

### Phase 1：确认目标

向用户确认审查目标，了解被审查系统的当前设计。如果已有 plan 文件，先读 plan。

### Phase 2：并行启动红蓝双方

同时启动两个后台 Agent（`run_in_background: true`）：

**红方 (Explore agent)**：
```
你是"红方攻击者"。审查以下系统设计，找出所有缺陷、边界情况和潜在问题。

攻击角度：
1. 功能正确性：核心逻辑是否有漏洞
2. 边界场景：极端输入、空数据、超大/超小规模
3. 数据一致性：不同模块间的数据格式、映射、同步
4. 性能与成本：冗余调用、瓶颈、资源浪费
5. 用户体验：错误提示、进度反馈、参数冲突
6. 可维护性：硬编码、耦合点、扩展困难

每条问题包含三个要素：
- 现象：用户或开发者能看到什么不对劲
- 原因：代码/设计层面哪里出了问题
- 影响：不修的后果是什么

按角度分组输出。不写修法（交给蓝方）。
```

**蓝方 (general-purpose agent)**：
```
你是"蓝方防守者"。先浏览项目代码了解现有架构，然后等待红方输出。

针对每条问题给出：
- 修法：具体怎么改，改哪个文件的哪个函数
- 改动量：大概几行代码
- 优先级：高/中/低（考虑影响面+改动成本）

防守原则：
- 方案务实，能在现有架构上改
- 优先低成本高收益
- 能改 prompt 就不改代码结构
- 能加一行规则就不调 LLM

末尾加 Top 10 快速收益清单（最有价值的 10 条）。
```

### Phase 3：汇总

红蓝双方完成后，主代理汇总为计划文件，写入项目 `plans/` 目录。

**计划格式要求**（重要）：

1. **快速收益 Top 10**：每条必须包含
   - 现象（用户看到了什么）
   - 原因（代码层面根因）
   - 修法（具体改哪里、怎么改）
   - 改动量（估算行数）

2. **完整改进清单**：按模块分组，表格列出 问题 / 原因 / 修法 / 改动量

3. **分批执行建议**：第一批最关键的几项，后面逐批扩展

## 注意事项

- 红方用 Explore agent 即可（不需要写代码），蓝方用 general-purpose（需要读代码找修复点）
- 如果审查范围是本项目的完整系统，蓝方需要 Read 所有核心文件
- 最终计划放在项目 `plans/` 目录下，用有意义的文件名
