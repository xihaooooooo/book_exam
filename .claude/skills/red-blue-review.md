# Red-Blue Design Review Skill

## 触发方式

用户说 "red-blue review" / "红蓝对抗" / "红蓝审查" / "攻防审查" 时启用。

## 审查对象

**审查的是设计文档（plan），不是代码。** 如果有对应的 plan 文件，先读 plan，围绕设计方案展开攻击和防守。代码只作为理解现有架构的参考——蓝方读代码是为了判断修法在现有架构上是否可行，不是为了审查代码本身。

## 执行流程

### Phase 1：确认目标

向用户确认审查目标。如果已有 plan 文件，先读 plan，向用户简述待审查的设计。

### Phase 2：并行启动红蓝双方

同时启动两个后台 Agent（`run_in_background: true`）：

**红方 (Explore agent)**：
```
你是"红方攻击者"。审查以下系统设计，找出所有缺陷、边界情况和潜在问题。

先阅读设计文档（plan），再读相关代码文件了解现有架构（作为判断缺陷严重度的参考，不是审查对象）。

攻击角度：
1. 功能正确性：核心逻辑是否有漏洞
2. 边界场景：极端输入、空数据、超大/超小规模
3. 数据一致性：不同模块间的数据格式、映射、同步
4. 性能与成本：冗余调用、瓶颈、资源浪费
5. 用户体验：错误提示、进度反馈、参数冲突
6. 可维护性：硬编码、耦合点、扩展困难

每条问题包含三个要素：
- 现象：用户或开发者能看到什么不对劲
- 原因：设计层面哪里出了问题（是计划里的哪个决策导致）
- 影响：不修的后果是什么

按角度分组输出。不写修法（交给蓝方）。
```

**蓝方 (general-purpose agent)**：
```
你是"蓝方防守者"。审查设计文档（plan），并读相关代码文件了解现有架构的约束。

先理解设计方案，再读代码中与此方案相关的现有模块，然后主动预判红方会攻击的点。

针对你预判的每个问题给出：
- 修法：在计划里怎么改（改哪个设计决策、加什么约束、补什么细节）
- 改动量预估：修进计划大概几行描述
- 优先级：高/中/低（考虑影响面+改动成本）

防守原则：
- 方案务实，能在现有架构上改
- 优先低成本高收益
- 能改 prompt 就不改代码结构
- 能加一行规则就不调 LLM

末尾加 Top 10 快速收益清单（最有价值的 10 条修法及其进入计划的改动量）。
```

### Phase 3：汇总审查报告

红蓝双方完成后，主代理汇总为**审查报告**，写入项目 `plans/` 目录。

审查报告格式：

```markdown
# 红蓝对抗审查报告：[方案名称]

> 审查目标：[plan 文件路径]

## 致命缺陷
[现象 / 原因 / 修法 / 优先级 / 改动量]

## 高风险
[...]

## 中等风险
[...]

## 低风险
[...]

## 分批执行建议

### 第一批：堵致命缺陷（~X 行计划描述）
### 第二批：修高风险（~X 行计划描述）
### 第三批：补中等风险（~X 行计划描述）
```

### Phase 4：更新原计划（关键步骤）

1. 将审查报告给用户过目
2. 向用户确认：**要不要按审查结果更新计划？**
3. 用户同意后，将审查发现的所有待修复项写入原 plan 文件，形成新版本
4. 计划更新完后，**停下来等用户确认计划**。不要自动开始写代码
5. **只有用户明确说"开始做"/"动手"/"修"之后，才按计划写代码**

## 核心规则

- **审查的是设计，不是代码。** 红蓝双方都围绕 plan 展开，代码只是上下文参考
- **审查完之后必须先更新计划，再写代码。** 不能审查完直接跳去改代码——计划是设计文件，必须是修改代码前的最后一份权威描述
- **最终审查报告放在项目 `plans/` 目录下**，用有意义的文件名
- **红方用 Explore agent**（不需要写代码），**蓝方用 general-purpose**（需要读代码找架构约束）
