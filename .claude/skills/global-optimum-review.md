# 全局最优审查 Skill

## 触发方式

用户说 "全局最优审查" / "全局最优" / "全局视角" / "不要局部最优" 时启用。

## 适用场景

- 设计一个新功能/模块，已经有了初步方案
- 正在两个实现方案之间做选择
- 感觉某个方案在局部很美但隐约担心全局影响
- 评审别人的设计方案

## 核心原则

**局部最优 ≠ 全局最优。** 每个模块都把自己做到极致，系统整体反而可能变差——耦合复杂、数据分裂、用户体验割裂、短期省事但长期堵路。

审查的目的不是直接否定方案，而是找出"局部看起来对、全局可能有问题"的决策点，给出替代思路。

## 执行流程

### Phase 1：确认审查目标

向用户确认：
- 要审查的功能/方案是什么
- 当前的设计思路（用户口述或读取已有 plan 文件）
- 用户特别担心的维度（可选）

### Phase 2：五维度审查

对方案逐一检查以下五个维度：

**1. 数据一致性**
- 这个模块单独存一份数据是不是最方便？
- 引入新数据源后，和已有数据源会不会分裂？
- 同一份数据会不会在多个地方以不同格式存在？
- 更新一个数据源时，其他地方能不能同步？

**2. 耦合方向**
- 为了让这个模块简单，是不是把复杂度推给了调用方？
- 调用方需要多知道多少"不该知道的事"？
- 抽象层级是否一致（不要在一个地方暴露高层语义、另一处暴露底层细节）？

**3. 用户/调用方认知负担**
- 功能内部逻辑很统一，但暴露出去的接口/规则和项目其他地方一致吗？
- 有没有引入一种"例外规则"，用户需要单独记住？
- 参数命名、返回值格式、错误提示风格是否和现有保持一致？

**4. 重复建设**
- 自己造一遍最可控，但项目里是不是已经有类似机制？
- 有没有现成的工具函数、表结构、Agent 可以复用？
- 复用现有机制的成本（改现有 vs 新造）分别是多少？

**5. 短期 vs 长期**
- 现在只做一步最省事，但会不会堵死后续扩展的路？
- 方案假设的条件（数据量、用户数、模块数）扩大 10 倍后还成立吗？
- 后续有计划中的功能依赖当前这个决策吗？

### Phase 3：输出审查报告

报告格式：

```markdown
# 全局最优审查报告

## 审查目标
[一句话描述被审查的方案]

## 发现的问题

### [问题标题]
- **局部视角下的合理性**：[为什么在当前模块看是对的]
- **全局视角下的问题**：[从整个系统看有什么隐患]
- **维度**：[数据一致性 / 耦合方向 / 认知负担 / 重复建设 / 短期vs长期]
- **严重程度**：[高 / 中 / 低]
- **替代思路**：[有没有更好的做法]

## 综合评估

- 全局风险等级：[高 / 中 / 低]
- 值得保留的设计：[列表]
- 建议重新考虑的设计：[列表]
- 如果没有更好的替代方案，至少应该补充什么约束或后续计划？
```

### Phase 4：更新计划（关键步骤）

1. 将审查报告给用户过目
2. 向用户确认：**要不要按审查结果更新计划？**
3. 用户同意后，将审查发现的问题和替代思路写入原 plan 文件，形成新版本
4. 计划更新完后，**停下来等用户确认计划**。不要自动开始写代码
5. **只有用户明确说"开始做"/"动手"/"修"之后，才按计划写代码**

## 输出原则

- **不建议直接推翻**：指出问题但不全盘否定，优先给替代思路
- **允许权衡**：承认有些时候局部方案在约束下是最优解，但要明确标注风险并给出对冲措施
- **务实优先**：替代方案必须在现有架构上可行，不能凭空设计理想系统

## 核心规则

- **审查的是设计，不是代码。** 五个维度都围绕设计方案展开，代码只作为理解现有架构的参考
- **审查完之后必须先更新计划，再写代码。** 不能审查完直接跳去改代码——计划是设计文件，必须是修改代码前的最后一份权威描述
- 这个 skill 是**设计决策前的自检**，不是实现后的 review（后者用红蓝对抗）
- 如果用户还没有明确方案，先引导用户说清楚当前设计思路
- 如果审查发现风险很低、方案无明显全局问题，直接告知用户即可，不必硬凑问题
